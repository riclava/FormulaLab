# FormulaLab 多阶段开发计划

本文档根据 [draft.md](/Users/ricl/github.com/riclava/FormulaLab/docs/draft.md) 拆解 FormulaLab Web V1 的开发路线。V1 的核心目标不是做完整知识库平台，而是尽快验证一个强学习闭环：

```text
内置高质量公式集 -> 首次诊断 -> 今日复习 -> 错误补弱 -> 记忆钩子 -> 简单调度 -> 复习总结
```

## 0. 开发原则

- Review First：用户打开后优先进入今日复习，而不是 Dashboard。
- 单一知识域优先：V1 只打磨一个高质量公式集，默认建议先用“概率统计常用公式”作为 Alpha 内容域。
- 训练闭环优先：每次开发都优先服务“能不能完成一次有效复习”。
- AI 先做内容辅助：V1 用户侧不做 AI 聊天入口，AI 主要辅助生成解释、题目、误用点和联想候选。
- 记忆钩子服务主动回忆：提示默认隐藏，只在用户卡住时出现。

## 0.1 设计原型参考

[demo.tsx](/Users/ricl/github.com/riclava/FormulaLab/docs/demo.tsx) 可以作为 V1 执行过程中的交互原型参考，尤其适合参考以下内容：

- 首次诊断进入今日复习的冷启动路径。
- Review 卡片的状态流转：题目 -> 提示 -> 答案 -> 自评。
- Again/Hard 后进入公式详情和记忆钩子的补弱闭环。
- 公式详情中“什么时候用 / 常见误用 / 变量说明 / 例题 / 关联公式 / 记忆钩子”的信息组织。
- 复习结束后的总结结构和薄弱公式回看入口。

但 demo 不应作为生产代码或最终视觉规范直接照搬：

- 当前 demo 使用 mock data 和单文件状态管理，实际开发应拆分为路由、组件、服务和数据层。
- 视觉上可以保留专注、轻量、训练感，但应收敛过大的圆角、过多嵌套卡片和单一 slate/indigo 色调。
- 交互实现需要补齐可访问性、键盘操作、焦点状态、表单 label、减少动画偏好和响应式细节。
- `transition-all`、`window.location.reload()`、无语义的可点击元素等原型写法应在正式实现中替换。

## 14. 推荐实施顺序

1. 先做数据模型和种子内容。
2. 再做 `/review`，哪怕先跳过完整诊断。
3. 接着补 `/diagnostic`，让新用户能进入队列。
4. 然后做 Again/Hard 后的公式详情和记忆钩子。
5. 最后做总结、统计和公式列表。

这个顺序保证每一步都在强化核心闭环，而不是先堆页面。
