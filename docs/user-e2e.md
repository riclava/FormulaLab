# FormulaLab User E2E Map

本文档从 UI/UX 视角梳理 FormulaLab 的长程用户操作，并把它们整理成可执行、可追踪的 E2E 链路。

更新于 2026-04-23。本文已同步最近一轮页面优化，文中“当前剩余问题”只保留仍待处理的部分。

## 1. 范围

这里的“长程操作”指的是同时满足以下条件的任务：

- 跨越多个页面、面板或状态阶段；
- 会改写学习状态、内容状态或提示优先级；
- 会影响用户下一步行为；
- 更适合按完整任务设计，而不是按单个页面动作设计。

本文按两类角色拆分：

1. 学习者（learner）：围绕诊断、复习、补弱、联想、总结完成训练闭环。
2. 内容运营者（operator）：围绕内容辅助、草稿审核、种子内容维护支撑学习闭环。

## 3. 当前闭环结构

### 3.1 学习者主闭环

```text
/review 空态
-> /diagnostic
-> /review
-> Again / Hard 时进入补弱卡片或详情
-> /summary
-> 返回 /review 或 /review?mode=weak
```

这是 V1 最核心的验证闭环。

### 3.2 学习者辅助闭环

```text
/formulas
-> /formulas/[id]
-> /memory-hooks 或 /derivation
-> 回流 /review
```

这些链路用于补理解、补联想、补边界，不应抢主闭环的注意力。

### 3.3 内部运营闭环

```text
/content-assist
-> /content-assist/[slug]
-> 生成 / 编辑 / 保存
-> 审核通过
-> approved 内容包
-> 下一次 db:seed 吸收
```

它不是 learner 主流程，但会直接影响 learner 看到的题目、解释、关系和联想候选质量。

## 4. 已落地的关键优化

最近一轮已经完成的清理和优化，后续评审时不必再重复提出：

- `/review` 已补上训练来源标签、预计剩余时间、`今日复习 / 弱项重练` 切换。
- Again / Hard 已从“直接打开重面板”改成“先给最小恢复动作，再决定是否展开补弱 sheet”。
- `/summary` 已改成动态主 CTA，长期统计和指标已下沉折叠。
- `/formulas/[id]` 已支持来源感知返回和入口相关 quick actions。
- `/formulas` 已减轻扫描负担，不再把所有统计平铺成主信息。
- `/memory-hooks` 已支持默认提示选择、提示预览和三步式工作流。

## 5. E2E 明细

### E2E-01 冷启动诊断到首轮训练

**目标**  
让第一次进入系统的学习者在 1 分钟内完成冷启动，并直接开始第一轮复习。

**链路**

```text
/review 空态 -> /diagnostic -> 提交诊断 -> /review 第一题
```

**用户关注**  
“我现在先做什么，做完会得到什么？”

**当前剩余问题**

- 诊断仍然更像独立功能页，不够像 onboarding step。
- 用户看不到为什么会拿到这组题。
- 诊断结束后缺少“已生成今日训练单”的过渡反馈。

**下一步**

- 把诊断包装成 onboarding，而不是单独功能页。
- 增加题组来源说明。
- 在回流 `/review` 之前增加“弱项预览 -> 第一题”的交接层。

### E2E-02 今日复习主闭环

**目标**  
让回访学习者打开就能开练，并顺滑完成 recall -> hint -> answer -> self-assessment -> schedule。

**链路**

```text
/review -> 主动回忆 -> 请求提示 -> 显示答案 -> 自评 -> 下一题 -> 完成
```

**用户关注**  
“这题为什么在今天出现，我还剩多少，做完能不能干净收尾？”

**当前剩余问题**

- 已有原因标签和剩余时长，但节奏反馈仍偏粗，例如没有单题耗时参考。
- `稍后再练` 只有当前页内反馈，缺少可追踪的“已推迟项”状态。
- 页面文案仍会零星混入 Again / Hard 术语，系统语言还不够统一。

**下一步**

- 增加更细的节奏反馈。
- 增加已推迟项的可见状态。
- 全链路统一为中文行为标签，英文术语只留在算法层。

### E2E-03 Again / Hard 补弱闭环

**目标**  
让用户在失败后立即进入恢复路径，而不是只留下失败记录。

**链路**

```text
/review 自评 Again / Hard
-> 最小恢复动作卡片
-> 补弱 sheet 或详情
-> 继续下一题 / 加入稍后再练
```

**用户关注**  
“我只想知道刚才为什么错，不想被拉进一个重页面。”

**当前剩余问题**

- 补弱入口已经变轻，但展开 sheet 后信息密度仍然偏高。
- Again 与 Hard 虽然区分了焦点区域，但还没有按具体 weak point 细化默认动作。
- 补弱 sheet 已显示当前题号，但还缺少整轮连续节奏提示。

**下一步**

- 继续压缩 sheet 首屏信息密度。
- 根据 weak point 给出更细的默认补救动作。
- 在补弱流程里增加“这一轮还剩多久”的提示。

### E2E-04 复习后总结到下一步行动

**目标**  
让用户结束一轮训练后，立刻知道接下来只该做什么。

**链路**

```text
/summary -> 看本轮结果 -> 执行唯一下一步 -> 回到复习或详情
```

**用户关注**  
“今天算结束了吗，还是还差一个补弱动作？”

**当前剩余问题**

- 最弱公式快捷入口仍然偏静态，没有完全按 weak point 自适应。
- “今天已完成”的收尾感还可以更强。
- 本轮联想活动虽然下沉了，但对 learner 仍然不是高优先信息。

**下一步**

- 让最弱公式入口按 weak point 动态切到边界 / 误用 / 联想。
- 增强“今天可以停在这里”的状态感。
- 继续压低次要记录信息的视觉优先级。

### E2E-05 薄弱公式专项重练

**目标**  
让用户集中捞回 Again / Hard 公式，而不是重跑整轮 today queue。

**链路**

```text
/review?mode=weak -> 弱项队列 -> Again / Hard 继续补弱 -> 回流主复习或结束
```

**用户关注**  
“我只想练最弱的几条，不想重复劳动。”

**当前剩余问题**

- 顶部已经有模式切换，但 weak mode 的筛选规则还没写清楚。
- today queue 与 weak queue 的边界变清楚了，但结束后是否还要回主队列仍不够明确。

**下一步**

- 在 weak mode 头部写清选题规则与数量预期。
- 明确说明“这轮练完之后回到哪里”。
- 支持从 `/summary` 携带更具体的弱项集合进入。

### E2E-06 公式查找与详情补弱

**目标**  
让用户能快速定位某条公式，确认边界后回到训练。

**链路**

```text
/formulas -> 搜索 / 过滤 -> /formulas/[id] -> 看边界 / 误用 / 联想 -> 回流训练
```

**用户关注**  
“我不是来逛目录的，我是来确认一件事的。”

**当前剩余问题**

- 公式列表已经收紧，但移动端长卡片仍然略重。
- 详情页已支持来源感知返回，不过列表页的主动作还不够 source-aware。
- 结果量大时缺少分组或 pinned result 机制。

**下一步**

- 继续做移动端折叠策略。
- 在列表页更明确区分“继续补弱”与“仅查看边界”。
- 给大结果集增加分组或置顶结果。

### E2E-07 记忆钩子管理与提示配置

**目标**  
让用户把 AI 候选转成自己的联想，并让后续提示真的优先使用它。

**链路**

```text
/memory-hooks
-> 选公式
-> 保存 / 编辑 AI 候选或手写联想
-> 设为默认提示
-> /review 请求 hint 时优先出现
```

**用户关注**  
“我写的这条提示，下次真的会先出现吗？”

**当前剩余问题**

- 默认提示与提示预览已经落地，但 AI 候选仍缺少“为什么推荐这条”的解释。
- `刷新候选` 还是直接替换 / 合并，没有版本对比。

**下一步**

- 给 AI 候选增加轻解释。
- 给刷新候选增加版本感或 diff。
- 继续减少单页同时展开的任务密度。

### E2E-08 推导强化训练

**目标**  
帮助“会背但不会用”的用户重新连接公式来源与适用边界。

**链路**

```text
/derivation -> 先回忆推导线索 -> 显示推导 -> 下一条或打开详情
```

**用户关注**  
“我需要确认自己是否真的理解了，不是再看一遍答案。”

**当前剩余问题**

- 推导训练仍是孤立工具，没有自评，也没有调度反馈。
- 页面缺少筛选、分组和选择入口。
- `下一条` 已更明确，但整体仍然不像真正的 path。

**下一步**

- 增加简单自评。
- 支持 domain / weak-only / completeness 过滤。
- 把串行浏览改成更明确的待补队列。

### E2E-09 内容集 / 学习路径推进

**目标**  
让用户按内容集推进，而不是在公式之间完全孤立地跳转。

**链路**

```text
/paths -> 查看按 domain 分组的内容 -> 打开详情 -> 进入复习或弱项重练
```

**用户关注**  
“这组内容我现在到哪了，接下来该推进哪一条？”

**当前剩余问题**

- `/paths` 仍然更像按 domain 分组的 catalog，而不是真正的学习路径。
- 公式顺序主要按 subdomain + difficulty 排，不一定符合依赖关系。
- 缺少完成度、下一条推荐和“先学还是先练”的判断。

**下一步**

- 增加完成度、待练数、最弱节点。
- 基于 prerequisite / confusable / application_of 重排顺序。
- 继续把每组内容收敛成一个明确主动作。

### E2E-10 自定义公式接入训练闭环

**目标**  
让高级用户把自己的公式接入同一套训练系统。

**链路**

```text
/formulas/new -> 填写公式 -> 创建 -> 自动生成 3 类训练题 -> 进入详情或训练
```

**用户关注**  
“我不想一次填太多，但也不想生成太差的训练题。”

**当前剩余问题**

- 表单仍然是一口气长表单，门槛偏高。
- 还没有真正拆出“最低可用模式”和“高级补充模式”。
- 创建后没有训练题预览。

**下一步**

- 分两步创建：先最小可练，再高级补充。
- 创建后展示 3 类题预览。
- 继续把这个入口定位成 advanced feature，而不是前台主入口。

### E2E-11 内部内容辅助与审核

**目标**  
让内部运营者高效生成、编辑、审核公式草稿，并安全落入 seed。

**链路**

```text
/content-assist
-> /content-assist/[slug]
-> 生成 / 编辑 / 保存
-> 审核通过
-> approved 内容包待 db:seed 吸收
```

**用户关注**  
“我需要知道这份草稿和正式内容差在哪，审核动作又不能误发。”

**当前剩余问题**

- 编辑器仍是超长单页，缺少目录、锚点和差异视图。
- `重新生成建议` 仍然缺少版本感。
- `审核通过并写入 seed 包` 仍缺确认层或变更摘要。

**下一步**

- 增加 sticky section nav 和未保存变更提示。
- 审核前展示 change summary。
- 把 regenerate 改成新候选版本，而不是直接覆盖。

## 6. 跨链路的系统性问题

### 6.1 主闭环与辅助闭环边界还不够清楚

- `/review` 仍然是主入口，但辅助入口还会被感知成平级工具。
- 系统离“训练工作台”更近了，但仍有一点“工具集合”感。

### 6.2 动态主 CTA 还没有全局统一

- `/summary`、`/review`、`/memory-hooks` 已开始突出唯一主动作。
- `/diagnostic`、`/paths`、`/derivation` 还没有同样强的优先级引导。

### 6.3 补弱链路仍然偏重

- 当前已经从“直接打开重面板”变成“先给恢复动作”，但完整补弱仍然略重。
- 系统仍然需要持续追求“最小恢复动作”，而不是完整知识面板。

### 6.4 状态解释还不够可感知

- 前台已经补上了“为什么是这题”“哪条提示会优先出现”“为什么推荐这个下一步”。
- 诊断、学习路径、推导训练里的状态解释仍然偏弱。

### 6.5 自动化 E2E 仍偏 API

- 当前 `tests/e2e/critical-path.test.ts` 主要覆盖 API smoke。
- 它能证明服务链路存在，但还不能证明真实 UI 旅程顺滑。

## 7. 推荐的 UI E2E 测试清单

建议优先补这 6 条真实浏览器 E2E：

1. 新用户：`/review` 空态 -> `/diagnostic` -> `/review` 第一题。
2. 主复习：题目 -> hint -> answer -> Good -> 下一题。
3. 失败补弱：Again -> 补弱卡片 -> sheet -> 加入稍后再练 -> 回到队列。
4. 结束总结：完成 session -> `/summary` -> 打开弱项详情 -> 回到复习。
5. 钩子生效：在 `/memory-hooks` 新建 / 设置默认 -> 看到提示预览 -> 下一次 `/review` hint 中出现。
6. 内部运营：`/content-assist/[slug]` 编辑 -> 保存 -> 审核通过。

## 8. 产品优先级建议

如果只按 UI/UX 投入顺序排序，建议是：

1. 继续打磨 E2E-01、E2E-02、E2E-03、E2E-04。
2. 稳住 E2E-07，让 memory hook 真正形成“写下 -> 被提示 -> 觉得有帮助 -> 更愿意再写”的闭环。
3. 再增强 E2E-05、E2E-06、E2E-09 这些辅助链路。
4. E2E-10 与 E2E-11 继续按高阶 / 内部流处理，不要反过来污染 learner 主入口。
