# FormulaLab User E2E Map

本文档从 UI/UX 视角梳理 FormulaLab 当前实现里的所有长程操作，并把它们整理成可执行的 E2E 链路。

更新于 2026-04-22：本文已同步最近一轮界面优化，下面“当前不合理”里只保留仍待处理的问题，不再重复已经落地的改进项。

这里的“长程操作”指的是：

- 跨越多个页面或状态面板；
- 会改写学习状态、内容状态或提示优先级；
- 对用户的下一步行为有明显影响；
- 适合被设计成完整任务，而不是单个页面动作。

本文档按两类角色拆分：

1. 学习者（learner）：最终用户，围绕诊断、复习、补弱、联想、总结完成训练闭环。
2. 内容运营者（operator）：内部使用者，围绕内容辅助、草稿审核、种子内容更新维护系统内容。

## 2. 主闭环与辅助闭环

### 2.1 学习者主闭环

```text
/review 空态
-> /diagnostic
-> /review
-> Again/Hard 时进入补弱 sheet 或详情
-> /summary
-> 返回 /review 或 /review?mode=weak
```

这是当前产品最应该打磨的闭环，也是 V1 成败最直接的部分。

### 2.2 学习者辅助闭环

```text
/formulas
-> /formulas/[id]
-> /memory-hooks 或 /derivation
-> 回流 /review
```

这些流程帮助用户“补理解”和“补联想”，但不应抢占主闭环的注意力。

### 2.3 内部运营闭环

```text
/content-assist
-> /content-assist/[slug]
-> 生成 / 编辑 / 保存
-> 审核通过
-> approved 内容包
-> 下一次 db:seed 吸收
```

这不是 learner 主流程，但它决定 learner 看到的题目、解释、关系和联想候选质量。

## 3. 详细 E2E 链路

## E2E-01 冷启动诊断到首轮训练

### 目标

让第一次进入系统的学习者在 1 分钟内完成冷启动，不需要理解平台结构，就能开始第一轮复习。

### 起点 / 终点

- 起点：`/review`
- 终点：`/review` 中第一题 ready to answer

### 当前链路

```text
访问 /review
-> 发现没有学习状态，看到空态
-> 点击“前往首次诊断”
-> 进入 /diagnostic
-> 完成 3-5 道诊断题自评
-> 系统创建初始公式状态与 review queue
-> 跳回 /review
-> 开始第一题
```

### 用户心智

- 我今天先做什么？
- 不想先研究导航，只想尽快开始。
- 希望系统替我挑出最该练的内容。

### 当前不合理

- `/review` 空态已经收敛成“开始 1 分钟诊断”为主动作，但 `/diagnostic` 与 onboarding 之间仍有页面切换感。
- 诊断页默认固定知识域，用户仍然看不到“为什么是这组题”。
- 诊断完成后的回流仍是直接跳到 `/review`，缺少“你刚得到了哪些优先复习项”的过渡反馈。

### 可优化方向

- 在 `/review` 空态只保留一个主 CTA：“开始 1 分钟诊断”，其余入口折叠为次级链接。
- 把诊断包装成 onboarding step，而不是单独功能页，例如标题更强调“生成今日训练单”。
- 在诊断过程中显示轻量预期收益，例如“完成后会直接生成今天最该练的 3-5 条公式”。
- 诊断结果页增加可见的“弱项预览 -> 第一题预览 -> 开始训练”连续过渡，减少割裂感。

## E2E-02 今日复习主闭环

### 目标

让回访学习者打开后立刻知道今天该练什么，并顺滑完成一轮 recall -> hint -> answer -> self-assessment -> schedule。

### 起点 / 终点

- 起点：`/review`
- 终点：一轮 `study_session` 完成

### 当前链路

```text
进入 /review
-> 系统拉取 today queue
-> 显示题目与进度
-> 用户先回忆
-> 需要时点“给我一点提示”
-> 再点“显示答案”
-> 选择“想不起来 / 有点吃力 / 基本记住 / 很轻松”
-> 系统更新 nextReviewAt
-> 下一题
-> 完成后进入完成态并可去 /summary
```

### 用户心智

- 我今天还剩多少？
- 这题为什么会出现在今天？
- 我现在应该“先想”、还是“先看提示”？
- 做完之后我能不能快速收尾，而不是继续被页面打扰？

### 当前不合理

- 题头已经有“今天到期 / Again 回收 / 诊断薄弱”等原因标签，也显示了预计剩余时间，但还缺少更细的节奏反馈，例如单题耗时或当前是否慢于平时。
- `稍后再练` 已经有明确反馈，不过当前仍然是内联消息，没有跨页可追踪的“已推迟项”状态。
- 主卡片与总结页已经用中文行为标签为主，但部分说明文案仍会穿插 Again / Hard 术语，系统语言还可以再统一。

### 可优化方向

- 在题目头部增加轻量原因标签，例如“今天到期”“诊断薄弱”“Again 回收”。
- 在 session 顶部增加“预计还需 X 分钟”或“平均每题 Y 秒”的节奏反馈。
- 对“稍后再练”给出 toast 或内联提示，例如“已放到今天队尾，约 10 分钟后再见”。
- 全链路统一自评语言：对外以中文行为标签为主，英文术语只在算法/说明层出现。

## E2E-03 Again/Hard 补弱闭环

### 目标

让卡住的用户在答错后立刻进入恢复路径，而不是只有一次失败记录。

### 起点 / 终点

- 起点：`/review` 中点击 Again 或 Hard
- 终点：继续下一题，或把当前公式放回今日稍后再练

### 当前链路

```text
在 /review 完成自评
-> 选择 Again 或 Hard
-> 打开 ReviewRemediationSheet
-> 进入 FormulaDetailView 聚焦 use 或 anti-patterns
-> 查看适用条件 / 常见误用 / 记忆钩子 / 关系
-> 可创建或保存 hook
-> 可加入今日稍后再练
-> 继续下一题 或 打开完整详情页
```

### 用户心智

- 我不是想“阅读百科”，我只是想知道这题为什么刚才做错。
- 我愿意做一个最小恢复动作，但不想被拉进太重的学习分支。
- 如果这次恢复有效，我应该马上回到训练节奏。

### 当前不合理

- 当前已经拆成“先给最小恢复动作，再决定是否展开 sheet”的两层补弱，但打开 sheet 后的信息密度依然偏高。
- Again 与 Hard 的恢复建议虽然已区分焦点区域，但还没有根据具体 weak point 给出更细的默认动作。
- 补弱 sheet 已显示当前是第几题，但还缺少“这一轮大概还剩多久”的连续节奏提示。

### 可优化方向

- 把补弱拆成两层：先给一个 1-step 恢复卡片，再决定是否展开完整 sheet。
- 对 Again 与 Hard 采用不同强度的默认动作：Again 更偏 hook/误用，Hard 更偏适用边界。
- 在 sheet 顶部固定显示“补弱当前题：第 N 题 / 共 M 题，完成后回到下一题”。
- 给出一个更强的 primary CTA，例如“看完这一条就继续”，把补弱从阅读任务变成单动作任务。

## E2E-04 复习后总结到下一步行动

### 目标

让用户结束一轮训练后，不是看“报表”，而是立刻知道接下来做什么。

### 起点 / 终点

- 起点：`/summary`
- 终点：进入下一次明确动作，例如继续补弱、返回今日复习、进入弱项重练、打开记忆钩子

### 当前链路

```text
完成 /review
-> 点击“查看总结”
-> 进入 /summary
-> 查看最新 session、弱项、下一次建议时间、推荐动作、长期统计、联想活动
-> 点击“回到今日复习”或弱项入口
```

### 用户心智

- 这一轮到底练得怎么样？
- 我现在最应该做的一件事是什么？
- 今天结束了吗，还是还差一个补弱动作？

### 当前不合理

- `/summary` 已经改成动态主 CTA，并把长期统计和指标折叠到后面，但最弱公式的快捷入口仍然偏静态，没有完全按 weak point 自适应。
- 当前已经区分“可以停在这里”和“建议先补弱”，不过“今天已完成”的仪式感还可以更强，比如更明确的收尾态或连续打卡反馈。
- “本轮联想活动”虽然已经下沉，但仍然更像学习记录，不是所有 learner 都会主动展开看。

### 可优化方向

- 将 `/summary` 拆成两级：上半屏只保留“结果 + 下一步”，下半屏折叠统计。
- 根据 session 结果做动态 CTA：
  - Again/Hard 多：主 CTA 为“先补弱 1 条”；
  - 全部 Good/Easy：主 CTA 为“今天完成”；
  - 有新 hook：主 CTA 为“下次优先使用这条提示”。
- 把“长期统计”“V1 指标”下沉到二级面板或 learner 不默认展开的区域。
- 增加“本轮结束”状态感，例如明确提示“今天的主任务已完成”。

## E2E-05 薄弱公式专项重练

### 目标

让用户在不重跑整轮 today queue 的情况下，集中捞回 Again/Hard 公式。

### 起点 / 终点

- 起点：`/review?mode=weak`
- 终点：弱项回收完成，回流 `/review` 或 `/summary`

### 当前链路

```text
从 /summary、/review 空态或其他入口进入 /review?mode=weak
-> 系统生成 weak queue
-> 用户完成和 today queue 类似的复习动作
-> Again/Hard 时继续进入补弱
-> 完成后返回 normal review 或结束
```

### 用户心智

- 我不想重新练全部，只想把最弱的那几条捞回来。
- 我需要确认系统为什么把这些题算作 weak。

### 当前不合理

- `/review` 顶部已经加入 `今日复习 | 弱项重练` 切换，但 weak mode 的筛选规则仍然没有写在头部。
- today queue 与 weak queue 的边界比之前清楚，但用户仍然不容易判断“这次弱项重练练完之后是否还需要回主队列”。

### 可优化方向

- 在 `/review` 顶部加入显式 segmented control：`今日复习 | 弱项重练`。
- 在 weak mode 头部显示规则说明与数量预期，例如“共 4 条，按最近 Again/Hard 排序”。
- 支持从 `/summary` 一键携带“最弱公式”集合进入 weak queue，而不是统一 query param。

## E2E-06 公式查找与详情补弱

### 目标

让用户在需要查询、确认边界、回看误用时，能快速定位到某条公式并回到训练。

### 起点 / 终点

- 起点：`/formulas`
- 终点：`/formulas/[id]` 完成查看后回流到 `/review`、`/memory-hooks` 或 `/derivation`

### 当前链路

```text
进入 /formulas
-> 搜索 / 过滤 domain、difficulty、tag
-> 打开 /formulas/[id]
-> 查看 use / non-use / anti-patterns / hooks / examples / relations / derivation
-> 进入相关练习或回到 review
```

### 用户心智

- 我不是来“逛列表”的，我是来快速确认一件事。
- 我想尽快知道这条公式什么时候该用、不该用。
- 如果我来自 review，我需要明确的返回路径。

### 当前不合理

- 公式列表已经压缩成“标题 + 状态 + 用途 + 下一次复习 + 少量标签”为主，但移动端长卡片仍然比较多，适合继续做折叠策略。
- 详情页已经带上下文返回和入口相关 quick actions，不过列表页还没有做到更强的 source-aware 主动作，例如直接区分“继续补弱”与“仅查看边界”。
- 标签和统计信息已经收敛，但大 domain 下结果很多时，缺少真正的分组或 pinned-result 机制。

### 可优化方向

- 公式列表默认只显示最关键的 3 个维度：标题、状态、one-line use，其余折叠。
- 从复习/总结深链进入详情时，增加 breadcrumb 或 sticky back action。
- 详情页 quick actions 根据入口智能切换：来自 Again 时先给 anti-patterns，来自 Paths 时先给 derivation/use。
- 对 tag 和统计行做折叠/截断策略，优先服务快速检索。

## E2E-07 记忆钩子管理与提示配置

### 目标

让用户能把 AI 候选转成自己的联想，管理个人 hook，并让后续提示真的优先使用它。

### 起点 / 终点

- 起点：`/memory-hooks` 或 `/formulas/[id]?focus=hooks`
- 终点：hook 保存、编辑、删除、设为默认提示，并在后续复习中被使用

### 当前链路

```text
进入 /memory-hooks
-> 选择某条公式
-> 查看个人联想与 AI 候选
-> 直接保存 AI 候选 / 编辑后保存 / 暂不采用
-> 或手动创建个人联想
-> 在复习时请求 hint
-> 系统优先使用被选中的 default hook
-> 用户可标记“这个提示有帮助”
```

### 用户心智

- 哪条联想最适合我？
- 我写的这条提示，真的会在下次优先出现吗？
- AI 候选和个人联想之间是什么关系？

### 当前不合理

- 默认提示选择和“复习中的提示预览”已经落地，承诺与实现基本对齐。
- 当前页已经加了三步式提示，但 AI 候选仍然缺少“为什么推荐这条”的解释。
- `刷新候选` 仍然是直接替换/合并列表，缺少版本对比，用户很难判断新一轮候选到底更好在哪里。

### 可优化方向

- 在 `/memory-hooks` workspace 中显式开启默认提示选择，并展示“当前默认”状态。
- 增加 preview 卡片：模拟在 review 中点击“给我一点提示”后会看到什么。
- 把页面改成三步结构：选公式 -> 选/写联想 -> 设为默认，而不是所有控件同时展开。
- 为 AI 候选增加“为什么推荐这条”的轻解释，降低保存阻力。

## E2E-08 推导强化训练

### 目标

服务“会背但不会用”的用户，让他们通过推导重新连接公式来源与适用边界。

### 起点 / 终点

- 起点：`/derivation`
- 终点：打开详情进一步理解，或回流 `/review?mode=weak`

### 当前链路

```text
进入 /derivation
-> 系统取有 derivation 的公式
-> 显示当前公式表达式与一句话用途
-> 用户先自己回忆推导线索
-> 点击“显示推导”
-> 点击“下一条”或打开完整详情
-> 回到弱项重练
```

### 用户心智

- 我需要先想一下这条公式从哪里来。
- 这次练完之后，系统应该知道我“理解更深了一点”。

### 当前不合理

- 推导训练是孤立工具，没有自评、没有记录“这次看懂了没”，也没有调度反馈。
- 页面没有筛选、分组、选择入口，只能串行下一条，控制感较弱。
- `下一条` 实际是轮换，不是明确推进，和用户预期的 path 感不一致。

### 可优化方向

- 增加简化自评：`还不懂 / 基本理顺 / 已能复述`，让 derivation 成为有状态的学习动作。
- 提供按 domain / weak-only / with-derivation completeness 的过滤。
- 将 `下一条` 改成更明确的 path 控件，例如“下一条待补推导公式”。

## E2E-09 内容集/学习路径推进

### 目标

让用户按知识域推进，而不是在公式之间完全孤立地跳转。

### 起点 / 终点

- 起点：`/paths`
- 终点：进入 `/review`、`/review?mode=weak` 或 `/formulas/[id]`

### 当前链路

```text
进入 /paths
-> 查看 domain 分组
-> 查看组内公式状态与顺序
-> 打开详情
-> 进入今日复习或弱项重练
```

### 用户心智

- 我想按一组内容往前推进，而不是只做系统随机分配的题。
- 我需要知道“这组内容我现在到哪了”。

### 当前不合理

- 当前 `/paths` 更像按 domain 分组的 catalog，不像真正的 path。
- 公式顺序主要按 subdomain + difficulty 排，不一定符合学习依赖。
- 页面缺少“完成进度”“下一条推荐”“这组该先练还是先学”的判断。

### 可优化方向

- 将 `/paths` 明确改成 content set progression，增加完成度、待练数、最弱节点。
- 用 prerequisite / confusable / application_of 关系生成更真实的 path 顺序。
- 每个 group 只保留一个主 CTA：`继续这组`，减少“查看内容集 / 进入今日复习 / 只练薄弱公式”并列竞争。

## E2E-10 自定义公式接入训练闭环

### 目标

让高级用户把自己的公式接入同一套训练系统。

### 起点 / 终点

- 起点：`/formulas/new`
- 终点：公式创建成功，自动生成题目并加入训练队列

### 当前链路

```text
进入 /formulas/new
-> 填写标题、表达式、用途、meaning、derivation、条件、误用、例题、tags、memory hook
-> 提交
-> 系统创建公式并自动生成 3 类训练题
-> 公式加入训练队列
-> 用户跳到详情或回到 /review
```

### 用户心智

- 我想把自己的内容也练起来。
- 我不想一口气填太多字段，但也不希望生成的题很差。

### 当前不合理

- 表单字段很多，且是一次性长表单，对普通用户门槛高。
- 没有“最低可用模式”和“高级补充模式”的层级。
- 提交后自动生成训练题是强能力，但页面没有题目预览，用户不清楚进入训练的质量如何。
- 这个入口与 V1 主航向并不一致，更适合高级/后期能力，而不是前台默认能力。

### 可优化方向

- 拆成两步：先创建最小公式，再补全解释/误用/例题。
- 提交后先展示“将生成的 3 类题预览”，再让用户选择立即进入训练。
- 把入口定位成 advanced feature，而不是 learner 常规入口。

## E2E-11 内部内容辅助与审核

### 目标

让内部运营者高效生成、编辑、审核公式解释、题目、关系与联想候选，并安全落入 seed。

### 起点 / 终点

- 起点：`/content-assist`
- 终点：某条公式的草稿被审核通过，写入 approved 包，等待 seed 吸收

### 当前链路

```text
进入 /content-assist
-> 浏览工作台列表
-> 打开 /content-assist/[slug]
-> 重新生成候选 / 编辑 explanation / review items / relations / memory hooks / reviewer notes
-> 保存草稿
-> 审核通过
-> approved 内容包待 db:seed 吸收
```

### 用户心智

- 我需要快速判断这份 AI 草稿能不能进正式内容。
- 我需要知道“当前草稿 vs 现有正式内容”的差异在哪里。
- 审核动作应该足够稳，不能一键误发。

### 当前不合理

- 编辑器是超长单页，缺少目录、锚点和差异视图，运营成本高。
- `重新生成建议` 可能覆盖当前心智上下文，但页面没有明显的 diff 或版本感。
- `审核通过并写入 seed 包` 是强动作，但缺少确认层或变更摘要。
- 没有 side-by-side 对比“当前 seed 内容 vs 草稿建议”，审核更像编辑，而不是 review。

### 可优化方向

- 增加 sticky section nav 和未保存变更提示。
- 审核前展示 change summary：解释改了哪些字段、题目改了哪些条目、关系和联想有哪些增删。
- 把 regenerate 变成“生成新候选版本”，允许与当前草稿对比后采纳。
- 审核动作增加确认弹层或 preflight checklist。

## 4. 跨链路的系统性问题

以下问题不是某一页独有，而是多个 E2E 会重复遇到的 UX 结构性问题。

### 4.1 主闭环与辅助闭环的边界还不够清楚

- `/review` 是主入口，但很多辅助入口仍然用平级导航出现。
- 对首次用户和普通回访用户来说，系统仍有一点“工具集合”感，而不是“训练工作台”感。

### 4.2 动态主 CTA 已开始出现，但还没有全局统一

- `/summary`、`/review`、`/memory-hooks` 已经开始按当前状态突出唯一主动作。
- 但 `/diagnostic`、`/paths`、`/derivation` 还没有同样强的“现在先做这一件事”引导。

### 4.3 补弱链路信息量偏大

- Again/Hard 后的补弱内容很完整，但对刚失败的用户来说有点重。
- 系统更需要的是“最小恢复动作”，而不是立刻提供完整知识面板。

### 4.4 学习状态解释还不够可感知

- 系统已在后台维护 queue、weak mode、nextReviewAt、memory hook helpfulness。
- 前台已经补上了部分解释，尤其是“为什么是这题”“哪条提示会优先出现”“为什么推荐这个下一步”。
- 但诊断、学习路径、推导训练里的状态含义仍然不够可感知。

### 4.5 自动化 E2E 覆盖偏 API，缺少真实 UI 旅程

- 当前 `tests/e2e/critical-path.test.ts` 主要覆盖 API smoke path。
- 它能证明服务链路存在，但不能证明真实用户在 UI 中能顺滑完成长程操作。

## 5. 推荐的 UI E2E 测试清单

后续如果要把这些长程操作落到真正的浏览器 E2E，建议先覆盖以下 6 条：

1. 新用户：`/review` 空态 -> `/diagnostic` -> `/review` 第一题。
2. 主复习：题目 -> hint -> answer -> Good -> 下一题。
3. 失败补弱：Again -> sheet -> 保存 hook -> 加入稍后再练 -> 回到队列。
4. 结束总结：完成 session -> `/summary` -> 打开弱项详情 -> 回到复习。
5. 钩子生效：在 `/memory-hooks` 新建/设置默认 -> 看到提示预览 -> 下一次 `/review` hint 中出现。
6. 内部运营：`/content-assist/[slug]` 编辑 -> 保存 -> 审核通过。

## 6. 产品优先级建议

如果只按 UI/UX 投入顺序排序，我建议：

1. 先打磨 E2E-01、E2E-02、E2E-03、E2E-04。
2. 再打通 E2E-07，让 memory hook 真正形成“写下 -> 被提示 -> 觉得有帮助 -> 更愿意再写”的闭环。
3. 之后再增强 E2E-05、E2E-06、E2E-09 这些辅助链路。
4. E2E-10 与 E2E-11 作为高阶/内部流，按角色分层处理，不要让它们反过来影响 learner 主入口的纯度。
